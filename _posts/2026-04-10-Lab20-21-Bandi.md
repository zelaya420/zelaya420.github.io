## Bandit – Nivel 20 → Nivel 21
### Objetivo del nivel

En este nivel, la contraseña del siguiente desafío no se obtiene leyendo un archivo directamente ni enviando datos a un servicio remoto conocido como en niveles previos.  
En su lugar, existe un programa llamado `suconnect` que **se conecta a un puerto local que tú le indiques**, espera recibir la contraseña actual del nivel y, si coincide, devuelve la contraseña del siguiente usuario.

La dificultad principal consiste en entender que aquí debes **levantar tu propio servicio local** que escuche en un puerto, enviarle la contraseña actual mediante `nc`, y hacer que `suconnect` se conecte a ese listener.

La sesión que compartiste muestra exactamente ese flujo: intentos iniciales con `nc -l`, luego un listener en background y finalmente la ejecución correcta de `./suconnect 12345`, que devuelve la contraseña de `bandit21`. :contentReference[oaicite:0]{index=0}

---

## Comandos útiles

Para completar este nivel, es posible que necesites apoyarte en los siguientes comandos:

- `nc` – Abrir listeners o conexiones TCP  
- `cat` – Visualizar el contenido de archivos  
- `echo` – Enviar texto a la salida estándar  
- `&` – Ejecutar un proceso en segundo plano  
- `./archivo` – Ejecutar un binario en el directorio actual  

En este nivel, los comandos clave son `nc`, `cat` y `./suconnect`.

## Conexión al servidor (Nivel 20)

Para acceder al Nivel 20 de Bandit, nos autenticamos por SSH con el usuario correspondiente a este nivel:

```javascript
sshpass -p '0qXahG8ZjOVMN9Ghs7iOWsCfZyXOUbYO' ssh bandit20@bandit.labs.overthewire.org -p 2220
```

Una vez dentro, el reto ya no consiste en buscar archivos visibles en el directorio personal, sino en hacer que un proceso local se conecte a otro proceso local.

¿Qué hace este nivel realmente?

El programa suconnect actúa como un cliente.

Su lógica es:

tú le dices un puerto
suconnect se conecta a localhost:ese_puerto
lee lo que reciba
si coincide con la contraseña actual de bandit20
responde con la contraseña del siguiente nivel

Eso implica algo muy importante:

tú debes preparar primero el servidor/listener que reciba la conexión.

¿Qué es un listener?

Un listener es un proceso que queda escuchando en un puerto, esperando que otro proceso se conecte.

En este nivel, ese listener lo levantas tú con nc.

Piensa en la arquitectura así:

nc -l -p 12345 → servidor/listener
./suconnect 12345 → cliente que se conecta a ese listener
Idea importante

Aquí tú controlas ambos lados del flujo:

el puerto que escucha
el dato que se envía al conectarse

Y el binario del nivel se encarga de validar la contraseña y devolver la siguiente.

Primer intento con nc

Probaste inicialmente:

echo "0qXahG8ZjOVMN9Ghs7iOWsCfZyXOUbYO" | nc -l -p 12345

pero luego lo interrumpiste con Ctrl+C. Esto se ve en tu sesión.

¿Por qué ese comando se quedaba “esperando”?

Porque nc -l -p 12345 abre un listener y se queda bloqueado hasta que alguien se conecte a ese puerto.

Eso es el comportamiento esperado.
No estaba “trabado”; estaba esperando al cliente.

Entonces, ¿por qué no avanzaba?

Porque mientras nc estaba ocupando la terminal en primer plano, aún no habías lanzado ./suconnect 12345 desde otro flujo de ejecución.

¿Por qué hacía falta un segundo proceso?

Aquí está la clave metodológica del nivel.

Si un comando se queda escuchando en la terminal actual, no puedes usar esa misma terminal al mismo tiempo para lanzar el cliente.

Necesitas una de estas dos cosas:

otra terminal/sesión
o mandar el listener al segundo plano con &

Tú elegiste la segunda opción, que fue correcta.

Solución correcta usando background

El comando que funcionó fue:

cat /etc/bandit_pass/bandit20 | nc -l -p 12345 &

La salida mostró:

[1] 22

Eso indica que el proceso quedó corriendo en segundo plano. Todo esto se observa en tu sesión.

Desglose del comando
1. cat /etc/bandit_pass/bandit20

Lee la contraseña actual del nivel 20.

2. |

La tubería envía esa salida al siguiente comando.

3. nc -l -p 12345

Abre un listener en el puerto 12345 y, cuando alguien se conecte, le entrega el contenido recibido por stdin.

4. &

Manda todo ese proceso al segundo plano.

Resultado lógico

El efecto es:

levantar un servidor local en 12345
que, cuando alguien se conecte, le envíe automáticamente la contraseña actual de bandit20
dejar libre la terminal para lanzar el cliente
Ejecución de suconnect

Con el listener ya activo, ejecutaste:

./suconnect 12345

La salida fue:

Read: 0qXahG8ZjOVMN9Ghs7iOWsCfZyXOUbYO
Password matches, sending next password
EeoULMCra2q0dSkYj561DX7s1CpBuOBt

Además, el proceso en background terminó después de cumplir su función:

[1]+  Done                    cat /etc/bandit_pass/bandit20 | nc -l -p 12345

Por tanto, la contraseña del usuario bandit21 es:

EeoULMCra2q0dSkYj561DX7s1CpBuOBt

Todo esto aparece directamente en la sesión compartida.

¿Qué hace exactamente ./suconnect 12345?

suconnect es un binario que actúa como cliente de red.

Cuando ejecutas:

./suconnect 12345

le estás diciendo:

conéctate a localhost
usando el puerto 12345
lee lo que el servicio te envíe
comprueba si coincide con la contraseña actual
si coincide, devuelve la siguiente contraseña
Observa el orden lógico

Primero debe existir el listener.
Luego se lanza suconnect.

Si haces esto al revés, el cliente intentará conectarse a un puerto donde nadie está escuchando.

¿Por qué usar cat /etc/bandit_pass/bandit20 y no echo?

También probaste con:

echo "0qXahG8ZjOVMN9Ghs7iOWsCfZyXOUbYO" | nc -l -p 12345

Eso conceptualmente podía funcionar, pero tiene un problema metodológico:
copiar la contraseña a mano aumenta el riesgo de error de transcripción.

La mejor solución fue:

cat /etc/bandit_pass/bandit20 | nc -l -p 12345 &

porque:

usa la contraseña exacta del archivo
evita errores humanos
automatiza la entrega al cliente
¿Qué significa -l -p 12345 en nc?
-l

Significa listen.
Le dice a nc que actúe como servidor y espere conexiones entrantes.

-p 12345

Indica el puerto en el que debe escuchar.

En otras palabras:

nc -l -p 12345

significa:

abre un listener
espera conexiones
hazlo en el puerto 12345
¿Qué significa &?

El símbolo:

&

manda el proceso al segundo plano.

Eso permite que la shell te devuelva el prompt y puedas seguir usando la terminal.

¿Por qué fue esencial aquí?

Porque necesitabas:

dejar corriendo el listener
lanzar después ./suconnect 12345

Sin &, el listener ocuparía la terminal y no podrías ejecutar el cliente en esa misma sesión.

¿Qué enseña realmente este nivel?

Este nivel enseña una idea muy importante en Linux y redes:

a veces no basta con conectarte a un servicio existente; debes crear tú mismo el servicio que otro proceso va a consumir.

La lógica del nivel entrena varios conceptos a la vez:

puertos locales
listeners
procesos cliente-servidor
ejecución en segundo plano
flujo de datos entre procesos
Modelo mental correcto del nivel

Piensa en este nivel así:

Tu listener con nc

Actúa como servidor temporal.

suconnect

Actúa como cliente que valida la contraseña.

Flujo completo
cat /etc/bandit_pass/bandit20
        |
        v
nc -l -p 12345    <----    ./suconnect 12345
        |
        v
si la contraseña coincide -> devuelve la siguiente
Error común en este nivel
1. Esperar que nc -l “termine solo”

No.
Un listener se queda esperando hasta que alguien se conecte.

2. Intentar hacerlo todo en primer plano

Si el listener ocupa la terminal, no podrás lanzar el cliente en la misma sesión.

3. Probar primero ./suconnect sin listener

Eso haría que el cliente no encontrara a quién conectarse.

4. Escribir la contraseña manualmente con echo

Puede funcionar, pero es peor práctica que leerla desde el archivo correcto.

Conclusión del nivel

Este nivel introduce varios conceptos fundamentales de redes y shell:

cómo crear un listener con nc
cómo enviar un secreto a través de un puerto local
cómo ejecutar procesos en segundo plano con &
cómo interactúan cliente y servidor dentro de una misma máquina
por qué el orden de ejecución importa

Más allá de resolver Bandit, este nivel enseña una idea muy útil:

los servicios de red no siempre están esperando por ti; a veces tú debes levantar el endpoint correcto para que otro proceso se conecte.
