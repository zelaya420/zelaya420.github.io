## Bandit – Nivel 23 → Nivel 24
### Objetivo del nivel

En este nivel, la contraseña del siguiente desafío no está guardada en un archivo visible ni en una ruta temporal fija como en el nivel anterior.  
La clave está en un **cron job** que se ejecuta automáticamente como `bandit24` y revisa un directorio especial:

```javascript
/var/spool/bandit24/foo
```

Si encuentra ahí un archivo normal, cuyo propietario sea bandit23, lo ejecuta y luego lo elimina.

La idea del nivel es aprovechar ese comportamiento: crear un script propio como bandit23, colocarlo en el directorio spool y hacer que, cuando el cron lo ejecute como bandit24, copie la contraseña de bandit24 a una ubicación que sí podamos leer.

Todo este flujo aparece reflejado en tu sesión.

Comandos útiles

Para completar este nivel, es posible que necesites apoyarte en los siguientes comandos:

cat – Visualizar contenido de archivos
chmod – Cambiar permisos de ejecución
cp – Copiar archivos
mktemp – Crear un directorio temporal aleatorio
ls – Listar archivos y directorios

En este nivel, los comandos clave son cat, chmod, cp y el análisis correcto del script del cron.

Conexión al servidor (Nivel 23)

Para acceder al Nivel 23 de Bandit, nos autenticamos por SSH con el usuario correspondiente:

sshpass -p '0Zf11ioIjMVN551jX3CmStKLYqjk54Ga' ssh bandit23@bandit.labs.overthewire.org -p 2220
Enumeración inicial del cron job

El primer paso correcto es revisar qué tarea automática está relacionada con bandit24:

cat /etc/cron.d/cronjob_bandit24

La salida fue:

@reboot bandit24 /usr/bin/cronjob_bandit24.sh &> /dev/null
* * * * * bandit24 /usr/bin/cronjob_bandit24.sh &> /dev/null

Esto nos dice dos cosas importantes:

el script corre como bandit24
se ejecuta cada minuto

Eso ya sugiere que no necesitamos explotarlo “en tiempo real”, sino preparar algo y dejar que el cron haga el resto.

Análisis del script real

Después revisamos el script que ejecuta el cron:

cat /usr/bin/cronjob_bandit24.sh

La salida fue:

#!/bin/bash

shopt -s nullglob

myname=$(whoami)

cd /var/spool/"$myname"/foo || exit 
echo "Executing and deleting all scripts in /var/spool/$myname/foo:"
for i in * .*;
do
    if [ "$i" != "." ] && [ "$i" != ".." ];
    then
        echo "Handling $i"
        owner="$(stat --format "%U" "./$i")"
        if [ "${owner}" = "bandit23" ] && [ -f "$i" ]; then
            timeout -s 9 60 "./$i"
        fi
        rm -rf "./$i"
    fi
done

Este script contiene toda la lógica del nivel.

Qué hace realmente este script
1. Se ejecuta como bandit24
myname=$(whoami)

Como el cron corre como bandit24, entonces whoami devuelve:

bandit24

Por eso el script entra en:

/var/spool/bandit24/foo

No entra en una ruta asociada a bandit23, sino al usuario con el que el cron está corriendo.

2. Recorre todos los archivos del spool
for i in * .*;
do

El script recorre:

archivos visibles
archivos ocultos

La línea:

shopt -s nullglob

evita problemas cuando no hay coincidencias de archivos.

3. Filtra lo que sí va a ejecutar

El criterio importante está aquí:

owner="$(stat --format "%U" "./$i")"
if [ "${owner}" = "bandit23" ] && [ -f "$i" ]; then
    timeout -s 9 60 "./$i"
fi

Esto significa que el cron solo ejecuta un archivo si:

el propietario del archivo es bandit23
es un archivo regular
y además no dura más de 60 segundos
Punto clave

Como tú estás logueado como bandit23, puedes crear exactamente ese tipo de archivo.

4. Después lo borra
rm -rf "./$i"

Esto significa que el script será:

ejecutado una vez
y luego eliminado automáticamente

Por eso este nivel no consiste en dejar algo permanente, sino en colocar un artefacto temporal que haga una acción rápida antes de desaparecer.

Estrategia correcta

La lógica correcta del nivel es:

crear un script como bandit23
hacer que ese script copie /etc/bandit_pass/bandit24 a un archivo en /tmp
darle permisos de ejecución
copiarlo a /var/spool/bandit24/foo/
esperar a que el cron lo ejecute
leer el resultado desde /tmp
Preparación de un directorio temporal

En tu sesión primero preparaste un directorio temporal de trabajo:

tmpdir=$(mktemp -d)
cd "$tmpdir"

Eso fue una buena decisión porque te permitió crear el script en un entorno limpio y controlado.

El script mínimo que había que crear

El script correcto y suficiente para resolver el nivel es:

#!/bin/bash
cat /etc/bandit_pass/bandit24 > /tmp/mi_pass_bandit24

La idea es muy simple:

el cron ejecuta este archivo como bandit24
por tanto, cat /etc/bandit_pass/bandit24 sí puede leer la contraseña
la redirige a /tmp/mi_pass_bandit24
luego tú, como bandit23, lees ese archivo temporal
Dar permisos de ejecución

Una vez creado el script, hay que hacerlo ejecutable:

chmod 755 b.sh

Aquí sí tiene sentido +x o un permiso como 755, porque estamos trabajando con un script Bash que debe ejecutarse, no con una clave privada como en niveles anteriores.

Copiar el script al directorio spool

Tu flujo práctico quedó encapsulado en un script como bb.sh, cuyo contenido era:

tmpdir=$(mktemp -d)
cd "$tmpdir"
cat > b.sh <<'EOF'
#!/bin/bash
cat /etc/bandit_pass/bandit24 > /tmp/mi_pass_bandit24
EOF
chmod 755 b.sh
cp b.sh /var/spool/bandit24/foo/

La idea de fondo es correcta: crear el script y colocarlo en el directorio que el cron revisa cada minuto.

Dos errores útiles que aparecieron en la sesión
1. El primer script tenía un EOF sobrante

En un intento previo, b.sh quedó así:

#!/bin/bash
cat /etc/bandit_pass/bandit24 > /tmp/mi_pass_bandit24
EOF

Ese EOF extra no debería formar parte del script final.
Es un residuo de haber construido el archivo con un heredoc.

2. No podías listar el directorio spool

Probaste:

ls -l /var/spool/bandit24/foo/

y recibiste:

ls: cannot open directory '/var/spool/bandit24/foo/': Permission denied

Eso no significa que la estrategia estuviera mal.
Solo significa que no tienes permisos de lectura/listado sobre ese directorio.

Idea importante

No necesitas poder listar el spool para poder aprovechar el cron.
Solo necesitas que el archivo llegue allí y que el cron lo ejecute.

Obtención de la contraseña del Nivel 24

Una vez que el cron ejecutó tu script, ya solo quedaba leer el archivo temporal que tu script había creado:

cat /tmp/mi_pass_bandit24

La salida fue:

gb8KRRCsshuZXI0tUuR6ypOFjiZbf3G8

Por tanto, la contraseña del usuario bandit24 es:

gb8KRRCsshuZXI0tUuR6ypOFjiZbf3G8

Esto aparece explícitamente en tu sesión.

Qué enseña realmente este nivel

Este nivel no trata solo de “meter un script en una carpeta”.

Lo que enseña de verdad es esto:

1. Un proceso automatizado puede convertirse en tu vector de acceso

No porque lo explotes arbitrariamente, sino porque entiendes exactamente qué entradas acepta y cómo las ejecuta.

2. El contexto de ejecución importa más que tu usuario actual

Tu script pertenece a bandit23, pero el cron lo ejecuta como bandit24.
Ese cambio de contexto es lo que vuelve útil el script.

3. La clave no era leer el secreto directamente

La clave era hacer que otro proceso con más privilegios lo leyera por ti y lo copiara a una ubicación accesible.

Error conceptual que debes evitar

El error más común aquí es pensar:

“Necesito entrar al directorio spool y controlarlo como si fuera mío”.

No exactamente.

Lo único que necesitabas entender era:

qué archivos ejecuta el cron
bajo qué condiciones
y qué acción mínima debía hacer tu script

Una vez entiendes eso, el resto es casi mecánico.

Conclusión del nivel

Este nivel introduce varios conceptos importantes de Linux y seguridad:

cómo analizar cron jobs que ejecutan archivos de otros usuarios
qué es un directorio spool
por qué importan el propietario y el bit de ejecución
cómo usar un script corto para hacer que otro usuario copie un secreto a una ruta temporal
cómo pensar en automatizaciones como superficies de interacción

La idea central del nivel es esta:

si comprendes exactamente cómo un proceso automatizado selecciona y ejecuta archivos, puedes diseñar una entrada válida para que ese proceso trabaje a tu favor.
