## Bandit – Nivel 13 → Nivel 14
### Objetivo del nivel

En este nivel, la contraseña del siguiente desafío no se obtiene leyendo un archivo de texto convencional.  
En su lugar, el usuario `bandit13` dispone de una **clave privada SSH** llamada `sshkey.private`, que debe utilizarse para autenticarse como el siguiente usuario.

La dificultad principal no está solo en identificar el archivo correcto, sino en comprender qué representa una clave privada, por qué debe protegerse y por qué necesita permisos estrictos para poder usarse correctamente con SSH.

---

## Comandos útiles

Para completar este nivel, es posible que necesites apoyarte en los siguientes comandos:

- `ls` – Mostrar el contenido de un directorio  
- `cat` – Visualizar el contenido de archivos  
- `chmod` – Cambiar permisos de archivos o directorios  
- `ssh` – Conectarse remotamente mediante SSH  

En este nivel, los comandos clave son `cat`, `chmod` y `ssh`.

## Conexión al servidor (Nivel 13)

Para acceder al Nivel 13 de Bandit, nos autenticamos por SSH con el usuario correspondiente a este nivel:

```javascript
sshpass -p 'FO5dwFsc0cbaIiH0h8J2eUks2vdTDwAn' ssh bandit13@bandit.labs.overthewire.org -p 2220
```

Una vez dentro, listamos el contenido del directorio:

ls

La salida muestra:

HINT  sshkey.private

Esto ya indica que el reto gira en torno a una clave privada SSH, no a una contraseña almacenada como texto plano.

Inspección del archivo sshkey.private

Al visualizar el contenido:

cat sshkey.private

la salida comienza con:

-----BEGIN RSA PRIVATE KEY-----
...
-----END RSA PRIVATE KEY-----

Eso identifica al archivo como una clave privada RSA en formato PEM.

¿Qué es sshkey.private?

sshkey.private es un archivo que contiene una clave privada SSH.

En SSH suele existir un par criptográfico:

clave privada: debe permanecer secreta
clave pública: puede compartirse

La autenticación funciona así, de forma simplificada:

el servidor tiene autorizada una clave pública
el cliente demuestra que posee la clave privada correspondiente
si ambas coinciden matemáticamente, el acceso es válido
Idea importante

La clave privada no es una contraseña normal escrita como texto plano.
Es una credencial criptográfica.

Eso implica que:

no se debe compartir
no debe quedar expuesta a otros usuarios
no conviene dejarla con permisos abiertos
¿Por qué este archivo se llama sshkey.private?

Aquí hay una precisión importante:
el nombre del archivo no es lo esencial.

Podría llamarse:

id_rsa
llave.pem
mi_clave
sshkey.private

Lo que realmente importa es el contenido interno del archivo, no su nombre.

En este caso, el nombre se eligió para que quede claro que se trata de una clave privada SSH, pero el sistema reconoce su naturaleza por el formato del contenido.

¿Por qué hay que darle chmod 600?

Este punto es crítico.

Una clave privada solo es segura si nadie más puede leerla.
Por eso se suele aplicar:

chmod 600 sshkey.private
¿Qué significa 600?

600 significa:

propietario: 6
grupo: 0
otros: 0

Cada dígito se calcula sumando:

4 = lectura (r)
2 = escritura (w)
1 = ejecución (x)

Entonces:

6 = 4 + 2 = rw-
0 = ---

Por tanto:

600 = rw-------

Eso significa:

el dueño puede leer y escribir
el grupo no puede hacer nada
otros no pueden hacer nada
¿Por qué SSH lo exige o lo prefiere?

Porque si una clave privada puede ser leída por otros usuarios, deja de ser realmente privada.

El cliente SSH suele rechazar claves demasiado abiertas con mensajes como:

Permissions 0644 for 'sshkey.private' are too open

La lógica es correcta: si otros pueden leer la clave, cualquiera podría intentar usarla.

Permisos más usados y para qué sirven
600
rw-------

Uso típico:

claves privadas SSH
archivos con secretos
configuraciones sensibles
400
r--------

Uso típico:

claves privadas que solo deben leerse
archivos muy sensibles que no necesitas editar
644
rw-r--r--

Uso típico:

archivos normales de texto
configuraciones que el dueño edita y otros pueden leer
700
rwx------

Uso típico:

directorios privados
scripts o binarios solo para el dueño
755
rwxr-xr-x

Uso típico:

scripts ejecutables
binarios
directorios accesibles por otros usuarios
¿Cómo se interpretan los permisos numéricos?

Cada bloque corresponde a:

owner (propietario)
group (grupo)
others (otros)

Y cada valor se forma así:

r = 4
w = 2
x = 1
Ejemplos
600 = rw-------
644 = rw-r--r--
755 = rwxr-xr-x
700 = rwx------
400 = r--------

La ventaja de la notación numérica es que fija el permiso de forma exacta.

Diferencia entre chmod 600 y chmod +x

Aquí se mezclan dos maneras diferentes de cambiar permisos.

Forma numérica
chmod 600 sshkey.private

Esto fija los permisos de forma exacta.

No añade “un poco”; reemplaza el estado completo por uno nuevo:

rw-------
Forma simbólica
chmod +x archivo

Esto no fija todo.
Solo añade permiso de ejecución (x) al archivo.

Diferencia conceptual
chmod 600 archivo = define exactamente qué permisos tendrá
chmod +x archivo = modifica solo una parte del permiso existente
¿Qué hace +x exactamente?

+x añade permiso de ejecución.

En archivos

Permite ejecutar el archivo como programa o script, por ejemplo:

chmod +x script.sh
./script.sh
En directorios

Permite “atravesar” el directorio, es decir, entrar con cd o acceder a su contenido si además hay permisos adecuados.

Pero en una clave privada SSH

No tiene sentido hacer:

chmod +x sshkey.private

porque una clave privada:

no es un script
no es un binario ejecutable
no debe ejecutarse
solo necesita ser leída por SSH

Por tanto, lo correcto es restringir su lectura, no darle ejecución.

Uso práctico de la clave privada

Una vez copiada a tu máquina y con permisos correctos:

chmod 600 sshkey.private

puede utilizarse con SSH así:

ssh -i sshkey.private bandit14@localhost -p 2220
¿Por qué -i?

La opción -i indica a ssh qué archivo de identidad, es decir, qué clave privada, debe usar para autenticarse.

Error común en este nivel

Un error frecuente es pensar que sshkey.private es simplemente “otro texto largo” o una contraseña rara.

No lo es.
Es una credencial criptográfica completa.

Otro error común es dejarla con permisos demasiado abiertos, por ejemplo:

chmod 755 sshkey.private

Eso sería un error serio, porque haría el archivo accesible de formas que no tienen sentido para una clave privada.

La lección aquí es clara:

una clave privada debe poder leerse por el dueño
debe estar protegida del resto
no necesita permiso de ejecución
Conclusión del nivel

Este nivel introduce varios conceptos fundamentales de Linux y SSH:

qué es una clave privada SSH
por qué una clave privada no debe compartirse
por qué el nombre del archivo es secundario frente a su contenido real
cómo funcionan los permisos en Linux
por qué chmod 600 es adecuado para una clave privada
cuál es la diferencia entre permisos numéricos y simbólicos

Más allá de resolver Bandit, este nivel enseña una idea central en sistemas y seguridad:

una credencial no solo debe existir; también debe almacenarse con permisos coherentes con su sensibilidad.

En otras palabras:
proteger un secreto no es solo tenerlo, sino controlar quién puede leerlo.
