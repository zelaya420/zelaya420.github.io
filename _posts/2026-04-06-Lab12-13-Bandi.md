## Bandit – Nivel 12 → Nivel 13
### Objetivo del nivel

En este nivel, la contraseña del siguiente desafío se encuentra en el archivo `data.txt`, pero no está almacenada de forma directa.  
El archivo contiene un **hexdump** de otro archivo que ha sido **comprimido repetidamente** en distintos formatos.

La dificultad principal consiste en entender que no basta con leer `data.txt`: primero hay que reconstruir el archivo binario original y, después, identificar correctamente cada capa de compresión o archivado hasta llegar al contenido final.

---

## Comandos útiles

Para completar este nivel, es posible que necesites apoyarte en los siguientes comandos:

- `mktemp` – Crear un directorio temporal aleatorio  
- `cp` – Copiar archivos  
- `mv` – Renombrar archivos  
- `xxd` – Convertir entre hexdump y binario  
- `file` – Identificar el tipo real de un archivo  
- `gzip` – Descomprimir archivos gzip  
- `bzip2` – Descomprimir archivos bzip2  
- `tar` – Extraer archivos tar  
- `cat` – Visualizar contenido de texto  

En este nivel, los comandos clave son `mktemp`, `xxd`, `file`, `gzip`, `bzip2` y `tar`.

## Conexión al servidor (Nivel 12)

Para acceder al Nivel 12 de Bandit, nos autenticamos por SSH con el usuario correspondiente a este nivel:

```javascript
sshpass -p '7x16WNeHIi5YkIhWsfFIqoognUTyj9Q4' ssh bandit12@bandit.labs.overthewire.org -p 2220
```

Una vez dentro, comprobamos que el archivo relevante es data.txt:

ls

Salida:

data.txt
¿Por qué se utiliza un directorio temporal?

Este punto es clave y no conviene tratarlo como una simple instrucción del nivel.

En este reto no basta con leer un archivo. Necesitamos:

copiar data.txt
renombrarlo varias veces
generar archivos nuevos como data.bin, data.gz, data.bz2, data.tar
extraer contenido con tar
descomprimir capas sucesivas

Eso significa que necesitamos un espacio de trabajo escribible.

Problema del directorio personal

En este nivel, el directorio home del usuario no permite escritura para este tipo de operaciones.
Eso se ve cuando intentamos hacer algo como:

mv data.txt data.gz

y obtenemos:

mv: cannot move 'data.txt' to 'data.gz': Permission denied

Por tanto, el problema no es el comando mv, sino el lugar donde se está intentando trabajar.

Entonces, ¿por qué /tmp?

/tmp se usa porque cumple justo lo que este nivel necesita:

es un área pensada para trabajo temporal
permite crear y modificar archivos
evita tocar el archivo original en el home
permite generar muchos archivos intermedios sin ensuciar el entorno principal
facilita reiniciar el proceso si algo sale mal
¿Por qué mktemp -d y no un nombre cualquiera?

Porque en un entorno multiusuario como Bandit, usar nombres predecibles en /tmp no es buena idea.

mktemp -d crea un directorio:

único
aleatorio
difícil de adivinar
menos propenso a colisiones o borrados accidentales

En otras palabras, no se usa /tmp “porque sí”, sino porque necesitamos un workspace aislado, desechable y escribible.

Creación del directorio temporal de trabajo

Creamos el directorio temporal y copiamos el archivo allí:

tmpdir=$(mktemp -d)
cp data.txt "$tmpdir"
cd "$tmpdir"

Este paso corrige dos errores comunes:

intentar trabajar directamente en ~
intentar usar $tmpdir antes de haberlo definido
Inspección inicial del archivo

Antes de transformar nada, conviene mirar el contenido de data.txt:

cat data.txt

La salida empieza así:

00000000: 1f8b 0808 10da cf69 0203 6461 7461 322e
00000010: 6269 6e00 0140 02bf fd42 5a68 3931 4159
00000020: 2653 59e1 71be e800 0018 7fff dec6 ff7c
00000030: bd9f 4fbf ff77 ffff bfed af5d bffb dffd

Esto no es el archivo comprimido en sí, sino un hexdump.

¿Qué es un hexdump?

Un hexdump es una representación textual de datos binarios en formato hexadecimal.
Sirve para inspeccionar archivos no legibles directamente, ver sus bytes y reconocer cabeceras.

Aquí la idea importante es esta:

data.txt no es todavía el archivo real que debemos descomprimir
data.txt es una representación en hexadecimal del archivo real

Por eso, el primer paso técnico verdadero es reconstruir el binario original.

Conversión del hexdump a binario

Renombramos el archivo y revertimos el hexdump con xxd:

mv data.txt data.hex
xxd -r data.hex > data.bin
¿Por qué xxd -r?
xxd normalmente convierte binario a hexadecimal
xxd -r hace lo contrario: toma el hexdump y reconstruye el archivo binario original

Luego verificamos el tipo real del archivo:

file data.bin

Salida:

data.bin: gzip compressed data, was "data2.bin", ...

Esto nos dice que la primera capa es gzip.

Primera capa: gzip

Como file identificó el archivo como gzip, seguimos con:

mv data.bin data.gz
gzip -d data.gz
file data

Salida:

data: bzip2 compressed data, block size = 900k

Aquí aparece la primera lección metodológica importante:

no hay que adivinar el siguiente formato.
Hay que volver a ejecutar file después de cada transformación.

Segunda capa: bzip2
mv data data.bz2
bzip2 -d data.bz2
file data

Salida:

data: gzip compressed data, was "data4.bin", ...

Ahora la nueva capa vuelve a ser gzip.

Tercera capa: gzip
mv data data.gz
gzip -d data.gz
file data

Salida:

data: POSIX tar archive (GNU)

Ahora ya no estamos ante un compresor, sino ante un archivo tar.

¿Qué es tar y para qué sirve?

Aquí conviene hacer una distinción importante:

gzip y bzip2 comprimen
tar normalmente empaqueta archivos, pero no necesariamente los comprime

Eso significa que cuando file dice POSIX tar archive, ya no toca usar gzip ni bzip2, sino extraer el contenido con tar -xf.

Cuarta capa: tar
mv data data.tar
tar -xf data.tar
ls

Salida:

data5.bin  data.hex  data.tar

Aquí apareció un archivo nuevo: data5.bin.

Quinta capa: otro tar

Continuamos inspeccionando:

mv data5.bin data5.tar
tar -xf data5.tar
ls
file *

Salida relevante:

data6.bin: bzip2 compressed data, block size = 900k

Es decir, ahora la nueva capa es bzip2.

Sexta capa: bzip2
mv data6.bin data6.bz2
bzip2 -d data6.bz2
file *

Salida relevante:

data6: POSIX tar archive (GNU)

Ahora el nuevo archivo vuelve a ser un tar.

Séptima capa: tar
mv data6 data6.tar
tar -xf data6.tar
ls
file *

Salida relevante:

data8.bin: gzip compressed data, was "data9.bin", ...

Otra vez aparece gzip.

Octava capa: gzip final
mv data8.bin data8.gz
gzip -d data8.gz
ls
file *

Salida relevante:

data8: ASCII text

Ahora sí, ya hemos llegado a un archivo de texto legible.

Obtención de la contraseña del Nivel 13

Solo queda leer el contenido final:

cat data8

Salida:

The password is FO5dwFsc0cbaIiH0h8J2eUks2vdTDwAn

La contraseña del usuario bandit13 es:

FO5dwFsc0cbaIiH0h8J2eUks2vdTDwAn
Errores comunes en este nivel
1. Intentar trabajar directamente en ~

Eso falla por permisos.
No es un error menor: es una pista estructural de que el home no es tu workspace.

2. Usar $tmpdir antes de definirlo

Si tmpdir no existe todavía, cp data.txt "$tmpdir" fallará.

3. Leer datos comprimidos con cat

Cuando hiciste:

cat data

la salida fue basura binaria.
Eso no significa que el archivo esté corrupto; significa que todavía no era texto final.

4. Seguir descomprimiendo “por intuición”

Cuando file dice bzip2, no debes insistir con gzip.
La herramienta correcta la dicta file, no la costumbre del paso anterior.

Conclusión del nivel

Este nivel enseña una metodología muy importante en Linux, forense y reversing básico:

trabajar en un directorio temporal cuando el original no es escribible
reconstruir binarios desde hexdumps con xxd -r
identificar el tipo real de archivo con file
distinguir compresión (gzip, bzip2) de archivado (tar)
repetir el proceso de forma iterativa hasta obtener texto legible

La enseñanza de fondo no es “descomprimir muchas veces”, sino esta:

no adivinar formatos; identificarlos y tratarlos según evidencia.

Además, el uso del directorio temporal no es un detalle decorativo:
fue la condición necesaria para poder manipular múltiples archivos intermedios sin romper el original ni chocar con restricciones de permisos.
