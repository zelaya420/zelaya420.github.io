## Bandit – Nivel 18 → Nivel 19
### Objetivo del nivel

En este nivel, la contraseña del siguiente desafío se encuentra en un archivo llamado `readme` dentro del directorio personal del usuario `bandit18`.

La dificultad principal consiste en que este nivel modifica el comportamiento del shell de inicio de sesión para impedir una sesión interactiva normal.  
Por ello, el objetivo no es “entrar y luego explorar”, sino **ejecutar directamente un comando remoto por SSH** antes de que el entorno problemático arruine la sesión.

---

## Comandos útiles

Para completar este nivel, es posible que necesites apoyarte en los siguientes comandos:

- `ssh` – Conectarse remotamente a otro sistema  
- `sshpass` – Pasar la contraseña a SSH de forma no interactiva  
- `cat` – Visualizar el contenido de archivos  

En este nivel, los comandos clave son `ssh` y `cat`.

---

## Conexión al servidor (Nivel 18)

Para acceder al Nivel 18 de Bandit, nos autenticamos por SSH con el usuario correspondiente a este nivel.

En lugar de abrir una sesión interactiva normal, ejecutamos directamente el comando remoto que nos interesa:

```javascript
sshpass -p 'x2gLTTjFwMOhQ8oWNbMN362QKxfRqGlO' ssh bandit18@bandit.labs.overthewire.org -p 2220 'cat readme'
```
La salida obtenida fue:

cGWpMaKXVwDUNgPAVJbWYuGHVn9zl3j8

Por tanto, la contraseña del usuario bandit19 es:

cGWpMaKXVwDUNgPAVJbWYuGHVn9zl3j8
¿Qué pasó realmente en este nivel?

Aquí el punto importante no es solo que readme contenga la contraseña.
El verdadero reto es que el entorno del usuario bandit18 está configurado para hacer incómoda o inútil una sesión interactiva normal.

Es decir:

conectarte “normalmente” por SSH puede no servir
el shell de login puede cerrarte la sesión o sabotearla
por eso conviene mandar el comando exacto desde el inicio
Idea importante

Este nivel cambia el modelo mental respecto a los anteriores.

Antes el patrón era:

entrar por SSH
enumerar
leer un archivo

Aquí el patrón correcto es:

conectarte por SSH
ejecutar directamente el comando remoto
recibir el resultado sin depender de un shell interactivo estable
¿Qué significa ejecutar un comando remoto por SSH?

Cuando escribes:

ssh usuario@host 'comando'

SSH no solo abre una conexión remota.
También puede pedirle al host remoto que ejecute un comando específico.

En tu caso:

ssh bandit18@bandit.labs.overthewire.org -p 2220 'cat readme'

significa:

conéctate como bandit18
usa el puerto 2220
en cuanto entres, ejecuta cat readme
devuelve la salida del comando
termina la conexión

Eso evita depender de una sesión interactiva completa.

Desglose del comando
sshpass -p 'x2gLTTjFwMOhQ8oWNbMN362QKxfRqGlO' ssh bandit18@bandit.labs.overthewire.org -p 2220 'cat readme'
1. sshpass -p '...'

Pasa la contraseña directamente al comando ssh sin que tengas que escribirla manualmente.

2. ssh bandit18@bandit.labs.overthewire.org -p 2220

Abre la conexión SSH al servidor Bandit usando el usuario bandit18 y el puerto correcto.

3. 'cat readme'

Este bloque le dice a SSH que ejecute ese comando en la máquina remota inmediatamente después de conectarse.

¿Por qué funciona aunque la shell sea problemática?

Porque SSH puede ejecutar un comando remoto sin que necesites quedarte trabajando dentro de una shell interactiva normal.

Ese detalle es justo lo que rompe el nivel.

Diferencia conceptual
Sesión interactiva normal
ssh bandit18@bandit.labs.overthewire.org -p 2220

Esto intenta dejarte dentro de un shell remoto para que interactúes manualmente.

Ejecución remota directa
ssh bandit18@bandit.labs.overthewire.org -p 2220 'cat readme'

Esto no necesita que te quedes navegando en la shell.
Solo le pide al servidor: “ejecuta esto y devuélveme la salida”.

¿Por qué usar comillas en 'cat readme'?

Las comillas agrupan el comando que debe ejecutarse remotamente.

Sin ellas, la shell local podría dividir mal la instrucción o interpretar partes del comando antes de enviarlo.

En este caso, usar:

'cat readme'

deja claro que eso es una sola instrucción remota.

¿Qué enseña realmente este nivel?

Este nivel enseña una idea muy útil en Linux y en administración remota:

SSH no solo sirve para abrir shells interactivas; también sirve para ejecutar comandos remotos de forma directa.

Eso es importante porque en sistemas reales muchas veces quieres:

ejecutar una sola orden
automatizar tareas
evitar una shell rota o restringida
recuperar información puntual sin iniciar una sesión completa
Error común en este nivel
1. Insistir en una sesión interactiva

Ese es el error más natural por costumbre de niveles anteriores.

Pero aquí el problema está justamente en el entorno interactivo.

2. Pensar que el nivel trata del archivo readme

No exactamente.
El archivo solo contiene la contraseña.
El verdadero reto es cómo leerlo cuando la sesión normal no coopera.

3. Olvidar que SSH puede ejecutar comandos remotos

Muchos usuarios asocian SSH únicamente con “entrar a otra máquina”.
Pero también es una herramienta excelente para lanzar comandos concretos sin abrir una sesión larga.

Alternativa equivalente

Si ya no quisieras usar sshpass, podrías hacer:

ssh bandit18@bandit.labs.overthewire.org -p 2220 'cat readme'

y escribir la contraseña cuando se te pidiera.

La lógica del nivel sería exactamente la misma.
sshpass solo automatiza la parte de autenticación.

Conclusión del nivel

Este nivel introduce un uso muy importante de SSH:

ejecutar comandos remotos directamente
evitar depender de shells interactivas
recuperar información puntual con una sola orden

Más allá de obtener la contraseña, este nivel enseña una idea muy útil en sistemas y seguridad:

cuando el entorno interactivo falla o estorba, no siempre necesitas una shell; a veces solo necesitas ejecutar el comando correcto desde el principio.
